<threat_model>
## Trust Boundaries

| Boundary | Description | Data Crossing |
|----------|-------------|---------------|
| AI review tool result -> renderer lock state | A newly created margin comment anchor becomes a temporary edit lock. | Serialized anchor payloads, runtime status |
| DOM selection -> edit guard | Browser selection/caret state is interpreted as allowed or blocked editing. | Range objects, input events, keyboard intent |
| Locked state -> user recovery | Lock ranges must disappear cleanly when review completes or fails. | Terminal runtime progress, cleared store state |

## Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Tampering | In-flight reviewed text | mitigate | Emit margin anchor ranges during AI review and prevent editing when the current selection overlaps those ranges. |
| T-02-05 | Denial of Service | Lock lifecycle | mitigate | Clear locks on terminal states so failed or completed reviews do not leave stale blocked passages behind. |
| T-02-06 | Integrity | DOM range overlap logic | mitigate | Extract pure overlap helpers and verify them with unit tests plus a live e2e locking scenario. |
</threat_model>
