<threat_model>
## Trust Boundaries

| Boundary | Description | Data Crossing |
|----------|-------------|---------------|
| Toolbar/popover -> Vuex retry state | The writer's selected persona and prompt must be preserved exactly for retry. | Persona choice and optional review prompt |
| Vuex runtime progress -> notification service | Failure state is turned into user-visible retry UI and must stay scoped to AI Review. | Runtime phase, status, error message |
| Retry confirm -> IPC review invocation | Confirming retry launches a new AI Review request and must not reuse stale data from another document. | Stored AI Review payload and current pathname |

## Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | Tampering | Renderer retry payload | mitigate | Store the exact last AI Review request before IPC invocation and clear it on document reset so retry cannot use the wrong persona or prompt. |
| T-01-05 | Denial of Service | Runtime-failure notification path | mitigate | Only show retry notifications for `PHASE_AI_REVIEW` failures to avoid duplicate retries across unrelated runtime jobs. |
| T-01-06 | Repudiation | Failure-state persistence | mitigate | Add a controller-level regression test proving partial comments remain after a failed review and the failed status still reaches renderer progress state. |
</threat_model>
